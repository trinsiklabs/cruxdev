# Computed Content Patterns

How projects in the CruxDev ecosystem derive content from code at build time, inject it into templates, verify it before deploy, and auto-update when source changes. This is the reusable pattern that BP110 (Metrics-Driven Content) implements for CruxDev specifically.

**Prerequisite:** Read `CONVERGENCE_PROCESS_IMPROVEMENTS.md` for the root cause analysis that motivated this pattern. Read `BIP_PATTERNS.md` for how generated content flows through the pipeline.

---

## 1. The Principle

If a number appears on a public page, it was computed from code by a deterministic function. Not typed by a human. Not generated by an LLM. Not copied from another file. Computed. From code. Every time.

This eliminates the entire class of stale-number bugs. The only way a number can be wrong is if the computation function is wrong, and that function has tests.

---

## 2. Architecture

```
Source Code (tests, tools, routes, dimensions, etc.)
    │
    ▼
Metrics Extractor (deterministic Rust function)
    │
    ▼
metrics.json (single source of truth, atomic write)
    │
    ├──▶ Astro templates (build-time import)
    ├──▶ llms.txt generator (template substitution)
    ├──▶ README updater (marker replacement)
    ├──▶ GTV Scanner (verification baseline)
    └──▶ BIP Pipeline (blog post Ground Truth section)
```

### 2.1 Data Flow

1. **Compute:** `compute_metrics(project_dir) -> MetricsPayload`
2. **Write:** Atomic write to `.cruxdev/metrics.json`
3. **Inject:** Build system reads `metrics.json`, populates templates
4. **Verify:** GTV scanner compares rendered pages against `metrics.json`
5. **Deploy:** Only if GTV passes

### 2.2 metrics.json Schema

```json
{
  "tests": 520,
  "tools": 65,
  "dimensions": 146,
  "dimension_sets": 21,
  "patterns": 14,
  "pages": 48,
  "blog_posts": 12,
  "vs_pages": 8,
  "binary_size_bytes": 5976064,
  "computed_at": "2026-03-28T12:00:00Z",
  "schema_version": 1
}
```

Every field is a concrete number. No strings describing numbers. No ranges. No approximations. The `computed_at` timestamp lets consumers detect staleness. The `schema_version` lets consumers reject incompatible formats.

## 3. The Generic Interface

### 3.1 Trait: `ComputedContent`

Every project that uses computed content implements this trait (conceptually; the actual implementation is a config-driven registry, not a Rust trait in every project):

```rust
/// A single computable metric.
pub struct MetricDef {
    /// Machine name: "tests", "tools", "dimensions"
    pub key: &'static str,
    /// Human description for error messages
    pub description: &'static str,
    /// How to compute the value
    pub compute: MetricCompute,
}

/// How a metric is computed.
pub enum MetricCompute {
    /// Count lines matching a pattern in command output
    CommandCount {
        cmd: &'static str,
        args: &'static [&'static str],
        count_pattern: &'static str,
    },
    /// Count files matching a glob
    GlobCount {
        pattern: &'static str,
        exclude: &'static [&'static str],
    },
    /// Count items in a Rust constant array (parsed from source)
    ConstantCount {
        file: &'static str,
        pattern: &'static str,
    },
    /// Stat a file
    FileSize {
        path: &'static str,
    },
    /// Call an MCP tool and count results
    McpToolCount {
        tool: &'static str,
        result_field: &'static str,
    },
    /// Custom function
    Custom(fn(&str) -> Result<u64, ComputeError>),
}
```

### 3.2 Project Declaration

A project declares its computed metrics in `.cruxdev/computed_content.toml`:

```toml
[metrics]
schema_version = 1
output = ".cruxdev/metrics.json"

[[metrics.definitions]]
key = "tests"
description = "Number of tests"
compute = "command_count"
cmd = "cargo"
args = ["test", "--list"]
count_pattern = ": test$"

[[metrics.definitions]]
key = "tools"
description = "Number of MCP tools"
compute = "mcp_tool_count"
tool = "tools/list"
result_field = "tools"

[[metrics.definitions]]
key = "dimensions"
description = "Number of audit dimensions"
compute = "constant_count"
file = "rust/src/engine/router.rs"
pattern = "DIMENSIONS"

[[metrics.definitions]]
key = "pages"
description = "Number of website pages"
compute = "glob_count"
pattern = "src/pages/**/*.{astro,md}"
exclude = ["node_modules"]

[injection]
# Where metrics are consumed
templates = [
    { path = "public/llms.txt.template", output = "public/llms.txt", format = "mustache" },
]

[injection.astro]
# Astro projects import metrics.json directly
data_file = "src/data/metrics.json"
type_file = "src/data/metrics.ts"

[injection.readme]
file = "README.md"
markers = { start = "<!-- metrics:start -->", end = "<!-- metrics:end -->" }
```

### 3.3 Auto-Detection

If a project has no `computed_content.toml`, CruxDev auto-detects common metrics:

| Project Type | Auto-Detected Metrics |
|---|---|
| Rust (Cargo.toml) | test count, binary size, clippy warnings |
| Python (pyproject.toml) | test count, coverage percentage |
| Node (package.json) | test count, build size |
| Astro site | page count, blog post count |
| Any with MCP server | tool count |
| Any with CruxDev | dimension count, pattern count |

Auto-detected metrics use convention-based defaults. Projects can override any auto-detected metric in `computed_content.toml`.

## 4. Injection Points

### 4.1 Astro (build-time import)

```typescript
// src/data/metrics.ts — generated, do not edit
import data from '../../.cruxdev/metrics.json';

export interface Metrics {
  tests: number;
  tools: number;
  dimensions: number;
  dimension_sets: number;
  patterns: number;
  pages: number;
  blog_posts: number;
  vs_pages: number;
  binary_size_bytes: number;
  computed_at: string;
  schema_version: number;
}

export const metrics: Metrics = data;
```

Usage in `.astro` files:

```astro
---
import { metrics } from '../data/metrics';
---
<h2>{metrics.tools} MCP Tools</h2>
<p>Verified by {metrics.tests} tests across {metrics.dimensions} audit dimensions</p>
```

### 4.2 Mustache Templates (llms.txt, etc.)

```
CruxDev is a convergence engine with {{tools}} MCP tools,
{{tests}} tests, and {{dimensions}} audit dimensions across
{{dimension_sets}} dimension sets.
```

The build script reads `metrics.json` and substitutes `{{key}}` markers.

### 4.3 README Markers

```markdown
<!-- metrics:start -->
| Metric | Value |
|--------|-------|
| Tests | 520 |
| MCP Tools | 65 |
| Audit Dimensions | 146 across 21 sets |
<!-- metrics:end -->
```

The `update-readme` script replaces everything between the markers with fresh values from `metrics.json`.

### 4.4 Blog Posts (special handling)

Blog posts do NOT use template variables. Blog posts are historical records. A blog post that says "520 tests" meant 520 tests on the day it was published. The correct handling is tiered:

| Blog Type | Handling |
|---|---|
| Milestone posts (`snapshot: true`) | Do not update. Add banner linking to live metrics. |
| Feature posts (incidental numbers) | Use relative language: "hundreds of tests", "dozens of tools" |
| Marketing pages (not blog) | Template variables. Always live. |

See `CONVERGENCE_PROCESS_IMPROVEMENTS.md` section 5 for the full blog strategy.

## 5. When to Recompute

### 5.1 Triggers

| Trigger | Mechanism | Latency |
|---|---|---|
| Pre-build (always) | `deploy.sh` runs `cruxdev metrics` before `npm run build` | Seconds |
| Post-convergence | CruxDev engine recomputes in PatternsUpdate phase | Immediate |
| Post-commit (CI) | GitHub Action runs `cruxdev metrics` on push | Minutes |
| Sentinel watch (CruxBot) | File watcher on test files, tool registration, router.rs | Seconds after change |
| Scheduled (CruxBot cron) | Every 4 hours as part of evolution cycle | Scheduled |

### 5.2 Staleness Detection

A metric is stale when:

```rust
fn is_stale(metrics_file: &Path, source_files: &[PathBuf]) -> bool {
    let metrics_mtime = fs::metadata(metrics_file)
        .and_then(|m| m.modified())
        .unwrap_or(SystemTime::UNIX_EPOCH);

    source_files.iter().any(|f| {
        fs::metadata(f)
            .and_then(|m| m.modified())
            .map(|t| t > metrics_mtime)
            .unwrap_or(true) // missing file = stale
    })
}
```

Source files that trigger staleness:

| Metric | Source Files |
|---|---|
| tests | `**/tests/**/*.rs`, `**/test_*.py`, `**/*.test.ts` |
| tools | `rust/src/server.rs`, any file registering MCP tools |
| dimensions | `rust/src/engine/router.rs` |
| pages | `src/pages/**/*.{astro,md}` |
| blog_posts | `src/pages/blog/**/*.md` |
| binary_size | `rust/target/release/cruxdev` |

### 5.3 What Happens When Stale

```
Staleness detected
    │
    ▼
Recompute metrics (deterministic, ~2 seconds)
    │
    ▼
Compare against rendered content (GTV cross-document consistency)
    │
    ├── All match → No action needed
    │
    └── Mismatch detected
         │
         ├── Marketing pages → Auto-rebuild + auto-deploy
         ├── README → Auto-update via marker replacement
         ├── llms.txt → Auto-regenerate from template
         └── Blog posts → Flag for review (do NOT auto-update)
```

## 6. Verification

### 6.1 GTV Integration

The GTV scanner's cross-document consistency checker uses `metrics.json` as the canonical baseline. When it finds a numeric claim like "520 tests" on a page, it compares against `metrics.json.tests`. If they disagree, the claim fails verification.

```rust
/// Verify a rendered page's numeric claims against computed metrics.
fn verify_against_metrics(
    page_content: &str,
    page_path: &str,
    metrics: &MetricsPayload,
) -> Vec<ConsistencyViolation> {
    let claims = extract_numeric_claims(page_content);
    let mut violations = Vec::new();

    for claim in claims {
        let canonical = match_claim_to_metric(&claim, metrics);
        if let Some((metric_key, expected)) = canonical {
            if claim.value != expected {
                violations.push(ConsistencyViolation {
                    file: page_path.to_string(),
                    line: claim.line,
                    claim: claim.text.clone(),
                    metric_key: metric_key.to_string(),
                    expected,
                    actual: claim.value,
                });
            }
        }
    }

    violations
}

/// Map a numeric claim to a metric key using keyword matching.
fn match_claim_to_metric(claim: &NumericClaim, metrics: &MetricsPayload) -> Option<(&str, u64)> {
    let text = claim.surrounding_text.to_lowercase();

    if text.contains("test") { return Some(("tests", metrics.tests)); }
    if text.contains("tool") { return Some(("tools", metrics.tools)); }
    if text.contains("dimension") && text.contains("set") {
        return Some(("dimension_sets", metrics.dimension_sets));
    }
    if text.contains("dimension") { return Some(("dimensions", metrics.dimensions)); }
    if text.contains("pattern") { return Some(("patterns", metrics.patterns)); }

    None
}
```

### 6.2 Deploy Gate

No deploy proceeds without fresh metrics verification:

```bash
#!/bin/bash
# deploy.sh — metrics-verified deploy

set -euo pipefail

# 1. Compute fresh metrics
cruxdev metrics --output .cruxdev/metrics.json

# 2. Copy to website data directory
cp .cruxdev/metrics.json ../cruxdev-dev/src/data/metrics.json

# 3. Build website (reads metrics.json at build time)
cd ../cruxdev-dev && npm run build

# 4. GTV verify rendered output against metrics
cd ../cruxdev && cruxdev gtv-scan-all ../cruxdev-dev/dist/ --verify-metrics .cruxdev/metrics.json

# 5. Deploy only if GTV passes
if [ $? -eq 0 ]; then
    rsync -avz ../cruxdev-dev/dist/ deploy@server:/var/www/cruxdev.dev/
else
    echo "DEPLOY BLOCKED: GTV verification failed"
    exit 1
fi
```

### 6.3 BIP Integration

When BIP generates a blog post after convergence, the Ground Truth section should read from `metrics.json`, not from the LLM's memory:

```
Generate a blog post about: [build plan title]
Ground Truth section MUST use these verified values:
- Tests: {metrics.tests} (from metrics.json computed at {metrics.computed_at})
- Tools: {metrics.tools}
- Dimensions: {metrics.dimensions} across {metrics.dimension_sets} sets
Do NOT use any other numbers for these metrics.
```

This is still an LLM call (content generation is inherently LLM), but the numeric facts are injected from the canonical source, not hallucinated.

## 7. Error Handling and Regression Detection

### 7.1 When Metrics Computation Fails

Metrics computation can fail (command not found, build broken, MCP server unresponsive). The failure mode matters:

| Failure | Impact | Response |
|---|---|---|
| `cargo test --list` fails | Test count unavailable | Block deploy. Do not use stale value. Fix the build first. |
| MCP server won't start | Tool count unavailable | Block deploy. The binary is broken. |
| Glob returns 0 results | Page/blog count is 0 | Warning, not blocker. Glob pattern may be wrong, or project has no pages yet. |
| `metrics.json` write fails | No metrics file produced | Block deploy. Disk full or permissions issue. |
| Single metric fails, others succeed | Partial metrics | Block deploy. Partial metrics.json is worse than none -- consumers assume all fields are present. |

The rule: metrics computation is all-or-nothing. Either every defined metric computes successfully and is written atomically, or the operation fails and no metrics.json is produced. Partial updates are never written.

### 7.2 Regression Detection

After computing new metrics, compare against the previous `metrics.json` (if it exists):

```rust
/// Detect metric regressions (values going DOWN unexpectedly).
fn detect_regressions(
    old: &MetricsPayload,
    new: &MetricsPayload,
) -> Vec<MetricRegression> {
    let mut regressions = Vec::new();

    // Test count should never decrease (unless tests are intentionally removed)
    if new.tests < old.tests {
        regressions.push(MetricRegression {
            metric: "tests".into(),
            old_value: old.tests,
            new_value: new.tests,
            severity: Severity::Warning,
        });
    }

    // Tool count decrease is a breaking change
    if new.tools < old.tools {
        regressions.push(MetricRegression {
            metric: "tools".into(),
            old_value: old.tools,
            new_value: new.tools,
            severity: Severity::Error,
        });
    }

    // Dimension count decrease means audit coverage loss
    if new.dimensions < old.dimensions {
        regressions.push(MetricRegression {
            metric: "dimensions".into(),
            old_value: old.dimensions,
            new_value: new.dimensions,
            severity: Severity::Warning,
        });
    }

    regressions
}
```

Regressions are logged but do not block deploy by default. A tool count decrease is flagged as `Severity::Error` because it indicates a potential breaking change for MCP consumers.

### 7.3 Audit Trail

Every metrics computation is logged to `.cruxdev/evolution/metrics_history.jsonl`:

```json
{"computed_at":"2026-03-28T12:00:00Z","tests":520,"tools":65,"dimensions":146,"trigger":"pre_deploy","duration_ms":1842,"regressions":[]}
```

This creates a time series of project metrics, useful for growth tracking and regression investigation.

## 8. Anti-Patterns

| Anti-Pattern | Why It Fails | Correct Approach |
|---|---|---|
| Partial metrics.json writes | Consumers assume all fields present; missing field causes template error | All-or-nothing atomic writes |
| Ignoring metric regressions | Test count dropping means bugs were introduced or coverage was lost | Log regressions, flag tool count drops as errors |
| LLM generates metric values | LLM hallucinates or uses stale context | LLM reads `metrics.json`; values are computed by code |
| Deploy without recomputing | metrics.json may be from last week | Always recompute before build |
| Cache metrics across convergence cycles | New code changes invalidate old metrics | Recompute from scratch every time |
| Template variable in blog posts | Blog posts are historical records; auto-updating creates citation instability | Use relative language or snapshot banners |
| Multiple metrics files | Drift between copies | One `metrics.json`, one `computed_content.toml` |
| Manual metric updates | Humans forget, numbers drift | Automated computation only |
| Approximate keywords in GTV matching | "520 tests and counting" matches "test" but "counting" is not a metric | GTV uses structured claim extraction, not substring matching |
| Metrics without `computed_at` timestamp | No way to detect staleness | Always include timestamp |
| Metrics without `schema_version` | Breaking changes to schema silently corrupt consumers | Version the schema; consumers reject unknown versions |
| Recomputing in the LLM call context | Adds latency to LLM calls, conflates concerns | Compute once in the build pipeline, read everywhere |

## 9. Cross-Project Pattern

### 9.1 How CruxBot Applies This to Any Project

When CruxBot adopts a project, the adoption checklist includes:

1. Does the project have public-facing numeric claims?
2. If yes, are those claims derived from code?
3. If no, create `computed_content.toml` with detected metrics
4. Wire metrics computation into the project's build pipeline
5. Wire GTV verification into the project's deploy pipeline

### 9.2 Per-Stack Implementation

| Stack | Metrics Computation | Template Injection | Build Integration |
|---|---|---|---|
| Rust + Astro (CruxDev) | `cargo run -- metrics` | `src/data/metrics.ts` import | `deploy.sh` pre-build step |
| Phoenix/Elixir (PETAL) | `mix test --list \| wc -l` + custom Mix task | EEx template assigns | `mix phx.digest` pre-step |
| Next.js | `jest --listTests \| wc -l` + build script | `src/data/metrics.json` import | `next.config.js` pre-build |
| Flutter | `flutter test --list \| wc -l` | N/A (mobile, no public pages) | CI only |
| Any with website | Project-specific extractor | JSON import or marker replacement | Pre-build script |

### 9.3 The Universal Flow

Regardless of stack, the flow is always:

```
1. compute_metrics(project_dir) -> metrics.json    [deterministic code]
2. inject_metrics(metrics.json, templates)          [build system]
3. verify_metrics(rendered_output, metrics.json)    [GTV scanner]
4. deploy_if_verified()                             [pipeline gate]
```

Steps 1, 3, and 4 are CruxDev-provided (via MCP tools or CLI). Step 2 is stack-specific but follows one of the patterns above.

## 10. Relationship to Existing Gates

### 10.1 GTV Scanner

The GTV scanner currently verifies individual claims against code reality. Computed content extends this by providing `metrics.json` as the authoritative baseline. The scanner no longer needs to re-run `cargo test --list` for every page it checks; it compares against the pre-computed, verified `metrics.json`.

### 10.2 Cross-Document Consistency Checker

`metrics.json` is the single source of truth. The consistency checker compares every file's claims against this one file. If two pages disagree, the one that matches `metrics.json` is correct; the other needs updating.

### 10.3 Pattern Adequacy Gate

A project implementing computed content does NOT need a METRICS_PATTERNS doc. This document (COMPUTED_CONTENT_PATTERNS.md) IS the pattern. The Pattern Adequacy Gate should recognize this file as sufficient for any build plan involving metrics, numbers, or computed content.

### 10.4 BIP Pipeline

BIP-generated blog posts should include a `metrics_snapshot` frontmatter field recording the metrics at time of publication:

```yaml
---
metrics_snapshot:
  tests: 520
  tools: 65
  dimensions: 146
---
```

This creates a permanent record of what the numbers were when the post was written, enabling future audits to distinguish "stale because code changed" from "wrong when published."

### 10.5 Convergence Engine

The convergence engine's `GtvVerification` phase (proposed in `CONVERGENCE_PROCESS_IMPROVEMENTS.md`) should include metrics freshness as a mandatory check:

1. Recompute `metrics.json`
2. Compare against all GTV targets
3. Block advancement on any mismatch

This is the same logic described in section 5.3, wired into the engine's phase progression.

## 12. The Standard

A number on a public page is either:
- Computed from code via `metrics.json` (marketing pages, README, llms.txt)
- A historical snapshot with a banner (blog posts)
- Relative language that does not need updating ("dozens of tools")

There is no fourth option. Hardcoded numbers in templates are a bug. This pattern eliminates that bug class permanently.
