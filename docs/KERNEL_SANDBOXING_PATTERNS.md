# Kernel-Level Sandboxing Patterns

**Gap:** Codex uses Apple Seatbelt on macOS, Landlock + seccomp on Linux.
**Priority:** Nice to have — CruxDev's rule-based safety gates handle most cases.

## macOS: Apple Seatbelt

```c
sandbox_init("(version 1)(deny default)(allow file-read*)(allow process-exec)", 0, &err);
```

Profile: deny network, deny write outside project dir, deny process execution outside allowlist.

## Linux: Landlock + seccomp

```rust
// Landlock (filesystem)
let ruleset = Ruleset::new(ABI::V5)
    .handle_access(AccessFs::from_all(ABI::V5))
    .create()?;
ruleset.add_rule(PathBeneath::new(project_dir, AccessFs::from_all(ABI::V5)))?;
ruleset.restrict_self()?;

// seccomp (syscalls)
// Allow: read, write, open, close, mmap, stat, etc.
// Deny: socket (no network), ptrace, mount
```

## CruxDev Current Safety

- 3-failure rollback
- 15-min timeout per task
- Net-negative detection
- Git branch isolation
- Never force push, never git add -A, reject secrets/binaries

## When Kernel Sandboxing Adds Value

- Running untrusted code generated by LLM
- CI/CD environments where isolation is critical
- Server-hosted CruxDev (not local dev)

## Recommendation

Nice to have for v2. Current rule-based gates are sufficient for local development. Kernel sandboxing becomes important when CruxDev runs in server/cloud context.
