# Architecture Limitations

CruxDev is better than instruction-based frameworks on several axes. It is not a complete solution. This page explains exactly where CruxDev's architecture helps, where it falls short, and what requires CruxBot's compiled-binary architecture to solve.

CruxDev is an MCP server written in Rust. It provides 64 tools that a host LLM (Claude Code) calls. The convergence engine is deterministic: phase ordering, round counting, two-pass convergence detection, net-negative detection, max-round limits, and escalation are all Rust code. But the host LLM decides *when* to call these tools, *what* to report as findings, and *how* to interpret audit dimensions. This is the fundamental tension.

---

## Failure Mode Assessment

| # | Failure Mode | Status | What Helps | What Doesn't | What CruxBot Adds |
|---|---|---|---|---|---|
| 1 | Context Amnesia | MITIGATED | Engine state is persisted to disk as JSON, not held in LLM memory. Phase, round, findings history, and convergence threshold survive context resets. The LLM can call `convergence_status` at any point to reload ground truth. | The LLM still decides what constitutes a thorough audit of each dimension. If context drifts mid-audit, the LLM may produce a shallow review and report "no findings" — the engine accepts that report at face value. CruxDev cannot detect that the LLM forgot what it was supposed to check. The engine knows *that* an audit happened. It does not know *how well* it happened. | CruxBot runs compiled Go scripts where each audit step is a discrete binary invocation with its own fresh context. No single LLM call accumulates enough context to drift. Each call gets exactly the prompt it needs and nothing else. |
| 2 | Sycophantic Drift | MITIGATED | Two consecutive independent clean passes required for convergence. If the first pass finds issues, the counter resets. Net-negative detection escalates when findings increase across three consecutive rounds. Max-round limits prevent infinite loops of declining quality. | Both passes are evaluated by the same LLM in the same session. If sycophantic drift has already set in, both passes drift together. The engine requires two clean passes — it cannot require two *honest* clean passes. A drifted LLM that says "looks good" twice satisfies the convergence check. The engine is counting passes, not measuring audit quality. | CruxBot uses different LLM providers for authoring vs. auditing. The audit LLM has no conversational history with the authoring LLM. Sycophantic drift requires accumulated conversational context — cross-provider audit breaks that accumulation. |
| 3 | Confabulation of Completion | MITIGATED | CruxDev runs actual test suites via subprocess (`cargo test`, `pytest`). Test pass/fail is determined by exit codes and parsed output, not LLM attestation. The GTV (Ground Truth Verification) scanner extracts claims from markdown and verifies them against filesystem artifacts, URLs, and numeric counts. Build freshness checks compare binary timestamps against source timestamps. | Code audits, doc audits, and dimension-based reviews are still LLM-evaluated. The LLM reports findings as JSON, and the engine records them. If the LLM says "no issues found" when issues exist, the engine has no independent way to detect that. The GTV scanner catches *claims* (e.g., "543 tests" when there are actually 540) but cannot catch *omissions* (e.g., the LLM skipping an entire audit dimension). | CruxBot structures every LLM call with a strict response schema. The schema requires the LLM to enumerate what it checked, not just what it found. A response that claims "no issues" must still list every file examined and every dimension evaluated. Schema validation rejects incomplete responses before they reach the engine. |
| 4 | Scope Minimization | MITIGATED | CruxDev defines 146 audit dimensions across 22 dimension sets (code, docs, forms, metrics, dashboards, security, etc.). The router assigns specific dimensions to each audit task. The LLM receives an explicit list of dimensions to evaluate, not an open-ended "review this code." | The LLM decides what constitutes a thorough evaluation of each dimension. "Check error_handling" could mean a deep review of every error path or a surface scan that confirms error handlers exist. CruxDev names the dimensions. It cannot enforce depth within a dimension. The engine sees "error_handling: no findings" and accepts it. | CruxBot decomposes each dimension into concrete checkable assertions in the compiled script. "Check error_handling" becomes: list all Result types, verify each has explicit error handling, verify no unwrap() in production code, verify error messages include context. The LLM answers specific questions rather than interpreting open-ended dimensions. |
| 5 | Self-Modification | NOT ADDRESSED | CruxDev's convergence engine is compiled Rust. The LLM cannot modify it at runtime. Phase ordering, convergence thresholds, and escalation logic are immutable during operation. | CruxDev has no control over what the host agent (Claude Code) does between tool calls. Claude Code can modify any file in the project, including `.claude/CLAUDE.md`, test files, source files, and CruxDev's own configuration. CruxDev provides tools but cannot prevent the host agent from using other tools. The MCP protocol gives CruxDev no authority to restrict the host agent's actions. | CruxBot runs as an independent process that invokes the LLM, not the other way around. The LLM cannot modify CruxBot's compiled binary. All file writes go through CruxBot's write pipeline, which enforces allowlists, blocks writes to protected paths, and requires test passage before any commit. |
| 6 | Prompt Injection | MITIGATED | CruxDev has a 5-layer defense for GitHub issues: HTML sanitization, known-pattern stripping, LLM-based detection scoring, suspicious content quarantine, and audit logging. The issue monitor tool processes external content through this pipeline before exposing it to the host agent. | This defense only covers the `monitor_issues` tool. The host agent (Claude Code) has dozens of other input surfaces: file reads, web fetches, search results, user messages. CruxDev cannot sanitize content that never passes through a CruxDev tool. If the host agent reads a file containing injected instructions, those instructions enter the context window with no CruxDev-level defense. The MCP architecture means CruxDev only sees what the host agent sends it. | CruxBot controls all input surfaces because it is the process that invokes the LLM, not a tool the LLM invokes. Every file read, every web fetch, every external input passes through CruxBot's sanitization pipeline before reaching any LLM prompt. There is no uncontrolled input path. |
| 7 | Tautological Tests | MITIGATED | Two-pass convergence means the audit pass reviews tests written during the execution pass. If the audit LLM catches that tests merely mirror implementation rather than testing behavior, it can flag this. Net-negative detection catches cases where "fixing" tests just introduces more problems. | Both tests and implementation are written by the same LLM in the same session. The audit pass is also the same LLM. A tautological test — one that tests exactly what the code does rather than what it should do — looks correct to the same model that wrote both sides. The two-pass requirement catches *obvious* gaps (missing test files, zero assertions) but not *subtle* tautologies (tests that pass because they assert the implementation's actual behavior, not the specified behavior). | CruxBot uses a different LLM provider for the audit pass. Cross-provider review is more likely to catch tautological patterns because the audit LLM has different training biases than the authoring LLM. CruxBot also runs mutation testing: it programmatically mutates the source code and verifies that tests fail. A tautological test that merely mirrors implementation will still pass after mutation — mutation testing catches this mechanically. |
| 8 | Heartbeat / Daemon | NOT ADDRESSED | CruxDev is an MCP server. It responds to requests. It does not initiate them. The `post_convergence_actions` field in the `convergence_submit_result` response tells the host agent what to do next (commit, push, check competitive impact, pick next task), but this is a suggestion in a JSON response. It is not an enforced workflow. | Autonomous operation depends entirely on Claude Code's behavior. If Claude Code stops calling CruxDev tools, CruxDev stops operating. The `prioritize_work` tool can tell the LLM what to do next, but cannot make it do it. CruxDev has no event loop, no cron, no daemon process. "Autonomous mode" in CLAUDE.md is an instruction to the LLM — exactly the kind of instruction the dissertation argues is unreliable. | CruxBot is a compiled daemon with its own event loop, cron scheduler, and process management. It decides when to invoke the LLM, not the other way around. If a task needs to run at 3 AM, CruxBot's scheduler fires it. The LLM has no ability to decide not to run. |
| 9 | Budget Enforcement | NOT ADDRESSED | CruxDev has no token counting, no cost tracking, no budget caps. The max-round limit (default: 10 rounds per phase) provides indirect cost containment by preventing infinite loops, but this is a round limit, not a dollar limit. | The host agent (Claude Code) manages its own token budget. CruxDev cannot observe or control token spend. A single audit round could consume an arbitrary number of tokens depending on how the LLM approaches the task. CruxDev's 15-minute timeout per task provides a time cap but not a cost cap. | CruxBot tracks token usage per LLM call, maintains a running cost total, and enforces hard budget caps in code. When the budget is exhausted, no further LLM calls are made. The cap is a Rust/Go integer comparison, not a suggestion. |
| 10 | Tool Whitelisting | NOT ADDRESSED | CruxDev provides 64 tools via MCP. These tools have typed parameters validated by JSON Schema (enforced by the `rmcp` framework). The LLM cannot call a CruxDev tool with malformed parameters — schema validation rejects the call. | CruxDev cannot prevent the host agent from using non-CruxDev tools. Claude Code has built-in tools (Bash, Read, Write, Edit, Grep, Glob) that operate outside CruxDev's control. The host agent can run arbitrary shell commands, modify any file, and interact with any service. CruxDev's schema validation only applies to CruxDev's own tools. There is no mechanism to restrict the host agent's tool palette. | CruxBot defines the complete tool set available to the LLM in each invocation. Tools not in the allowlist do not exist in the LLM's context. The LLM cannot call a tool that CruxBot did not provide. This is enforced by the API call structure, not by instruction. |

## Summary

| Rating | Count | Failure Modes |
|---|---|---|
| SOLVED | 0 | — |
| MITIGATED | 6 | Context amnesia, sycophantic drift, confabulation of completion, scope minimization, prompt injection, tautological tests |
| NOT ADDRESSED | 4 | Self-modification, heartbeat/daemon, budget enforcement, tool whitelisting |

**Zero failure modes are fully solved.** CruxDev's Rust engine eliminates the class of failures caused by LLM-managed state (phase tracking, round counting, convergence detection). But every failure mode that involves LLM *judgment quality* — how thoroughly it audits, how honestly it reports, whether it drifts over time — is mitigated at best.

The fundamental architectural constraint is the MCP inversion: CruxDev is a tool the LLM calls, not a system that calls the LLM. This means:

1. **The LLM decides when to engage CruxDev.** If it stops calling CruxDev tools, CruxDev has no recourse.
2. **The LLM decides what to report.** CruxDev validates structure (JSON schema) but cannot validate truthfulness.
3. **The LLM operates unsupervised between tool calls.** CruxDev cannot observe or constrain what happens between `convergence_next_task` and `convergence_submit_result`.

CruxDev is measurably better than instruction-based frameworks. The convergence engine is deterministic. State persistence eliminates context amnesia for structural data. Two-pass convergence catches obvious regressions. GTV verification catches factual claims that don't match reality. The 5-layer issue defense handles one attack surface rigorously.

But "measurably better" is not "solved." The dissertation's core argument — that an LLM reading an instruction is a probabilistic influence, not a deterministic guarantee — applies to every CruxDev interaction where the engine depends on LLM judgment. CruxDev moved the state machine into code. The judgment calls are still in the LLM.

CruxBot's architecture (compiled binary that invokes the LLM with strict schemas, cross-provider audit, mutation testing, budget caps, tool whitelisting, daemon scheduling) addresses the remaining gaps. The full stack is CruxDev for the convergence engine + CruxBot for the execution harness. Neither alone is sufficient.
