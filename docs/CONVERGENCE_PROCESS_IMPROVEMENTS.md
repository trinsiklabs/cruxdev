# Convergence Process Improvements — Post-External-Review Analysis

**Date:** 2026-03-28
**Triggered by:** External review found 14 issues, most critically stale numbers across 30+ pages, unsourced marketing claims, and cross-document inconsistencies. These are exactly the class of issues CruxDev's GTV system is designed to catch. The fact that they shipped to production means the convergence process has structural gaps.

---

## 1. Root Cause Analysis

### 1.1 Why stale numbers shipped

The convergence engine has 11 phases (10 active + Converged terminal). The phase sequence is:

```
Planning → PlanAuditing → DocAlignment → Viability → Executing →
CodeAuditing → DocAuditing → WebsiteConvergence → E2eTesting →
PatternsUpdate → Converged
```

The problem is what each phase actually checks:

| Phase | What it audits | What it misses |
|-------|---------------|----------------|
| CodeAuditing | Code correctness, completeness, security, test coverage | Numbers published about the code |
| DocAuditing | Doc accuracy, completeness, consistency | Whether doc numbers match code reality |
| WebsiteConvergence | Website accuracy, completeness, metrics, deployment | Cross-site consistency, number freshness |

The critical gap: **no phase runs the GTV scanner against live content**. The `WebsiteConvergence` phase (`router.rs` line 484) generates audit tasks with dimensions like "accuracy" and "completeness", but these are LLM-judged dimensions — the LLM reads the pages and evaluates them subjectively. There is no automated, deterministic check that the number "487 tests" on the homepage matches the output of `cargo test`.

The GTV scanner (`rust/src/gtv/scanner.rs`) exists as MCP tools (`gtv_scan_content`, `gtv_scan_all`) but is never called by the convergence engine. It is a tool the LLM can call, but the engine never routes a task that requires it. This is the core failure: **GTV is opt-in, not mandatory**.

### 1.2 Why cross-document inconsistencies exist

Numbers live in at least 6 independent locations:
- `cruxdev/README.md`
- `cruxdev-dev/public/llms.txt`
- `cruxdev-dev/src/pages/index.astro` (homepage)
- `cruxdev-dev/src/pages/engine.astro`
- `cruxdev-dev/src/pages/for/*.astro` (landing pages)
- `cruxdev-dev/src/pages/vs/*.astro` (comparison pages)
- `cruxdev-dev/src/pages/blog/*.md` (blog posts)

Each was written or updated at a different time. There is no single source of truth that feeds all of them. When a new tool is added (changing the count from 61 to 63 to 65), the developer updates the nearest file (usually README or the page they are working on) but not all the others.

### 1.3 Why the BIP pipeline did not catch this

BIP (Blog It, Post It) per `docs/BIP_PATTERNS.md` has a "Ground Truth" section requirement in blog posts. But:
1. The Ground Truth section is generated by the LLM, not by running `gtv_scan_content`.
2. The LLM generating the blog post uses whatever numbers it has in context, which may already be stale.
3. BIP fires on convergence completion but does not verify that previously published content still holds.

### 1.4 The temporal decay problem

Every convergence completion changes the ground truth. When BP067 adds 12 new tests, the test count on the homepage instantly becomes stale. But the homepage was not part of BP067's convergence — it was deployed in a previous build plan. No mechanism exists to detect that a previous deployment now contains stale data.

## 2. Missing Gates

### 2.1 Pre-deploy GTV gate (BUILD_PLAN_102, Phase 3.15 — incomplete)

BUILD_PLAN_102 correctly identified this need but items 3.15 and 3.16 remain unchecked:
- `3.15 Wire into deploy.sh: run gtv_scan_all on changed pages before rsync, block deploy on failures`
- `3.16 Wire into convergence post-actions: GTV scan required after content generation`

**This is the most critical missing gate.** The scanner exists. The MCP tools exist. They are just not wired into the deploy pipeline or the convergence engine.

### 2.2 Cross-document consistency gate

The GTV scanner verifies individual claims against code reality (e.g., "does `cargo test` produce 520 tests?"). What it does NOT do is verify cross-document consistency (e.g., "does the homepage say the same test count as README, llms.txt, and the blog?").

This is a different class of verification. The scanner would need to:
1. Extract all numeric claims from all public-facing documents
2. Group them by claim type (test count, tool count, dimension count)
3. Flag any group where the values disagree

### 2.3 Citation verification gate

The "~30% miss rate" claim and the "90.2% improvement" claim appear on multiple pages with no linked source. The GTV scanner extracts feature and status claims but does not check whether statistical claims have citations.

### 2.4 Staleness detection for deployed content

No mechanism detects that a previously deployed page has become stale. The convergence engine only operates on the current build plan's scope. If BP067 adds tests but BP067's scope does not include the website, the website pages are never re-scanned.

### 2.5 Canonical number registry

There is no single source of truth for project metrics. The numbers are hardcoded in each file. When the tool count changes, a human (or LLM) must manually update every file. This should be a computed value, derived from the code, stored once, and referenced everywhere.

## 3. Recommended Process Changes

### 3.1 New convergence phase: GtvVerification

Insert a new phase between `WebsiteConvergence` and `E2eTesting`:

```
... → WebsiteConvergence → GtvVerification → E2eTesting → ...
```

This phase is not LLM-judged. It is deterministic. The engine runs the GTV scanner against all public-facing files and blocks advancement if any high-confidence failure is detected.

New helper function:

```rust
/// Collect all files that need GTV verification before deploy.
fn collect_gtv_targets(proj_dir: &Path) -> Vec<String> {
    let mut targets = Vec::new();

    // README
    let readme = proj_dir.join("README.md");
    if readme.exists() {
        targets.push(readme.to_string_lossy().to_string());
    }

    // Sibling website project (-dev convention)
    let dev_dir = format!("{}-dev", proj_dir.to_string_lossy().trim_end_matches('/'));
    let pages_dir = Path::new(&dev_dir).join("src/pages");
    if pages_dir.exists() {
        // Scan all .astro and .md files in pages/
        collect_files_recursive(&pages_dir, &["astro", "md"], &mut targets);
    }

    // llms.txt
    let llms = Path::new(&dev_dir).join("public/llms.txt");
    if llms.exists() {
        targets.push(llms.to_string_lossy().to_string());
    }

    targets
}
```

### 3.3 Cross-document consistency checker

Add to `rust/src/gtv/consistency.rs`:

```rust
/// Check that all public-facing documents agree on key metrics.
pub fn check_cross_document_consistency(
    files: &[PathBuf],
    registry: &ProjectMetrics,
) -> Vec<ConsistencyViolation> {
    let mut violations = Vec::new();

    for file in files {
        let content = match std::fs::read_to_string(file) {
            Ok(c) => c,
            Err(_) => continue,
        };

        let claims = scanner::extract_claims(&content);
        for claim in &claims {
            if claim.claim_type != ClaimType::Numeric { continue; }

            // Parse "520 tests" -> (520, "tests")
            let (number, keyword) = parse_numeric_claim(&claim.value);

            let canonical = match keyword {
                k if k.contains("test") => Some(registry.test_count),
                k if k.contains("tool") => Some(registry.tool_count),
                k if k.contains("dimension") => Some(registry.dimension_count),
                _ => None,
            };

            if let Some(expected) = canonical {
                if number != expected {
                    violations.push(ConsistencyViolation {
                        file: file.to_string_lossy().to_string(),
                        line: claim.line_number,
                        claim: claim.value.clone(),
                        expected,
                        actual: number,
                    });
                }
            }
        }
    }

    violations
}
```

### 3.5 Temporal staleness detection

After every convergence that changes code metrics (adding tests, tools, or dimensions), the engine should:

1. Recompute `ProjectMetrics` via `ProjectMetrics::compute()`
2. Compare against the registry (`metrics.json`)
3. If any metric changed, run the cross-document consistency checker against all public-facing files
4. If inconsistencies found, emit a finding that blocks convergence until fixed

This should happen in the `PatternsUpdate` phase (the last phase before `Converged`), or in the new `GtvVerification` phase.

**Specific change in `router.rs`, PatternsUpdate arm (line ~592):**

```rust
ConvergencePhase::PatternsUpdate => {
    // Before checking convergence, verify metrics haven't drifted
    let proj_dir = resolve_proj_dir(state);
    let current = ProjectMetrics::compute(proj_dir.to_str().unwrap_or("."));
    let stale_files = check_cross_document_consistency(
        &collect_gtv_targets(&proj_dir),
        &current,
    );

    if !stale_files.is_empty() {
        // Do NOT converge — there are stale numbers in public content
        return Task {
            task_type: "fix_stale_metrics".into(),
            description: format!(
                "BLOCKED: {} files have stale metrics. Update all public-facing \
                 documents to match canonical values: {} tests, {} tools, \
                 {} dimensions across {} sets.",
                stale_files.len(),
                current.test_count,
                current.tool_count,
                current.dimension_count,
                current.dimension_set_count,
            ),
            files: stale_files.iter().map(|v| v.file.clone()).collect(),
            dimensions: vec!["stat_accuracy".into(), "consistency".into()],
            finding: Some(json!({"stale_metrics": stale_files})),
            test_command: None,
            metadata: Some(json!({"action": "update_stale_numbers", "canonical": current})),
```

## 4. CruxBot Enforcement

CruxBot is the autonomous agent that drives CruxDev. It needs these code-level gates:

### 4.1 Pre-deploy hook in CruxBot

CruxBot's deploy flow should call `gtv_scan_all` before executing any deploy command. If any high-confidence failure is detected, the deploy is blocked and a finding is filed.

**File: CruxBot's deploy module**
```rust
async fn pre_deploy_check(&self, project: &Project) -> Result<(), DeployBlocker> {
    let scan = self.mcp_call("gtv_scan_all", json!({
        "directory": project.website_dir(),
        "project_dir": project.root_dir(),
    })).await?;

    let result: GtvSummary = serde_json::from_str(&scan)?;
    if result.failed > 0 {
        return Err(DeployBlocker::StaleContent {
            failed_claims: result.failed,
            details: result.failures,
        });
    }
    Ok(())
}
```

### 4.2 Post-convergence metrics refresh

After any convergence completes, CruxBot should:
1. Recompute `ProjectMetrics`
2. Compare against all public-facing documents
3. If any document is stale, file a P0 issue or start a micro-convergence to fix it

### 4.3 Scheduled GTV sweep

CruxBot's cron should run a full GTV scan weekly (or on every convergence completion). This catches staleness from:
- External changes (competitor pages updated, URLs broken)
- Temporal decay (blog posts referencing old numbers)
- Cross-project drift (CruxDev metrics changed but CruxBot's docs still reference old numbers)

### 4.4 Deploy-blocking CI check

Add a CI workflow (`.github/workflows/gtv.yml`) that runs the GTV scanner on PR to the website repo:

```yaml
name: GTV Pre-Deploy
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  gtv:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install CruxDev binary
        run: curl -sSL https://cruxdev.dev/install.sh | bash
      - name: Run GTV scan
        run: cruxdev gtv-scan-all ./src/pages --project-dir ../cruxdev
      - name: Check results
        run: |
          if [ "$(cruxdev gtv-scan-all ./src/pages --format=json | jq .failed)" -gt 0 ]; then
            echo "GTV FAILED: stale claims detected"
            exit 1
          fi
```

## 5. Blog Post Strategy

Blog posts are the hardest problem because they are both historical records and marketing content.

### 5.1 The problem

When a blog post says "CruxDev has 485 tests," that was true when published. But a reader finding the post 3 months later sees a number that contradicts the homepage. This erodes trust — the exact thing CruxDev claims to prevent.

### 5.2 Recommended approach: tiered strategy

**Tier 1: Milestone posts (numbers are the point of the post)**
Examples: "485 tests: how we got here," "61 MCP tools"

These posts exist to celebrate a specific milestone. The number IS the content. Do NOT update these numbers. Instead:
- Add a frontmatter field: `snapshot: true`
- Render a visible banner: "This post is a snapshot from [date]. Current numbers: [live link to /stats or /engine]."
- The GTV scanner should skip `snapshot: true` posts for numeric staleness checks.

**Tier 2: Feature posts (numbers are incidental)**
Examples: "GitHub issue monitoring," "Crux ecosystem overview"

These posts mention numbers in passing ("CruxDev's 61 tools include..."). The number is not the point but becomes misleading when stale. Options:
- **Preferred:** Replace hardcoded numbers with relative language: "CruxDev's growing set of MCP tools includes..." or "With dozens of MCP tools..."
- **Acceptable:** Update the number when it drifts by more than 10%. Add a `last_verified: YYYY-MM-DD` field.
- **Not recommended:** Auto-update via template variables. Blog posts should be stable documents; auto-updating creates citation instability.

**Tier 3: Marketing landing pages (numbers are selling points)**
Examples: Homepage, /for/software-engineers, /vs/* pages

These MUST always show current numbers. These should use template variables populated from the canonical metrics registry, not hardcoded values.

### 5.3 Implementation for Tier 3

In the Astro site, create a data file that the build reads:

**File: `cruxdev-dev/src/data/metrics.json`** (generated by CI or pre-build script)

**File: `cruxdev-dev/src/data/metrics.ts`**

### 5.4 Scanner changes for blog posts

Add blog-awareness to the GTV scanner:

```rust
// In scanner.rs
pub fn extract_claims_with_context(content: &str, file_path: &str) -> Vec<Claim> {
    let is_blog = file_path.contains("/blog/");
    let is_snapshot = content.contains("snapshot: true");

    let mut claims = extract_claims(content);

    if is_blog && is_snapshot {
        // Mark all numeric claims as historical — skip verification
        for claim in &mut claims {
            if claim.claim_type == ClaimType::Numeric {
                claim.claim_type = ClaimType::HistoricalNumeric;
            }
        }
    }

    claims
}
```

### New convergence gates (summary)

1. **GtvVerification phase** — deterministic GTV scan of all public content, blocks advancement on high-confidence failures
2. **Staleness check in PatternsUpdate** — recomputes metrics, compares against public content, blocks convergence if stale
3. **Cross-document consistency** — all documents must agree on the same numbers for the same metrics
4. **Citation check** — statistical claims (percentages, rates) must have nearby source links
5. **Pre-deploy CI gate** — GTV scan in GitHub Actions, blocks merge/deploy on failure
6. **CruxBot weekly sweep** — scheduled full GTV scan to catch temporal decay

### Process change: the convergence contract

The current convergence contract is: "Two consecutive independent clean passes = converged."

The new contract adds: "AND all public-facing claims verified against code reality by the GTV scanner."

Clean passes are necessary but not sufficient. A round can be "clean" in the LLM's subjective judgment while containing objectively stale numbers that the LLM did not notice (because it does not have the current `cargo test` output in context). The GTV scanner closes this gap by providing deterministic verification that no LLM judgment can override.

## 7. Priority and Sequencing

| Priority | Change | Impact |
|----------|--------|--------|
| **P0** | Template variables in Astro marketing pages + `metrics.json` | Eliminates entire class of stale marketing numbers |
| **P0** | `count_all_dimensions()` in `router.rs` with test | Makes dimension count computable, not guessed |
| **P1** | GtvVerification phase in convergence engine | Prevents stale content from reaching Converged state |
| **P1** | Cross-document consistency checker | Catches disagreements between README, llms.txt, website |
| **P2** | Citation verification for statistical claims | Prevents unsourced claims from shipping |
| **P2** | Blog snapshot mechanism | Handles temporal decay in historical posts |
| **P3** | CI gate for website repo | Prevents stale deploys even when convergence is bypassed |
| **P3** | CruxBot weekly sweep | Catches external link rot and temporal decay |

The P0 items should be implemented first because they eliminate the root cause (hardcoded numbers in multiple locations). The convergence phase changes (P1) prevent regression. The P2-P3 items handle edge cases and external factors.
